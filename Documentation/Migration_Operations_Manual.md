# Multitenancy Migration Operations Manual

## 1. Overview
This manual is designed to guide the operations team in safely migrating the POS system from a single-tenant architecture to a multi-tenant architecture. The migration process involves database schema changes, data cleansing, and application updates.

**Risk Level**: High
**Estimated Downtime**: 2-4 hours (depending on data volume)

---

## 2. Pre-Migration Preparation

### 2.1 Data Backup
A full backup of the production database must be performed before executing any operations.

**SQL Server:**
```sql
BACKUP DATABASE [POSDb] TO DISK = 'C:\Backups\POSDb_PreMigration.bak' WITH FORMAT;
```

**SQLite:**
Directly copy the `.db` file to a secure directory.

### 2.2 Environment Check
*   Ensure all background tasks (Hangfire Jobs) are paused or completed.
*   Notify all users that the system will undergo maintenance.
*   Confirm that the target environment has .NET 6/7/8/10 Runtime installed (based on the actual project version).

---

## 3. Database Migration Steps

### 3.1 Create Default Tenant
In the new multi-tenant architecture, all existing data will be archived under a "Default Tenant".

```sql
-- SQL Server / SQLite compatible script
INSERT INTO Tenants (Id, Name, Host, IsActive, CreatedDate, CreatedBy, ModifiedDate, ModifiedBy, IsDeleted)
VALUES ('11111111-1111-1111-1111-111111111111', 'Default Tenant', 'localhost', 1, GETDATE(), '00000000-0000-0000-0000-000000000000', GETDATE(), '00000000-0000-0000-0000-000000000000', 0);
```

### 3.2 Schema Change
It is recommended to use migration scripts generated by EF Core. However, if manual SQL execution is required, the logic is as follows:

1.  **Add Column**: Add `TenantId` column to all business tables, initially allowing NULL.
    ```sql
    ALTER TABLE Products ADD TenantId UNIQUEIDENTIFIER NULL;
    ALTER TABLE Users ADD TenantId UNIQUEIDENTIFIER NULL;
    -- Repeat for all tables...
    ```

2.  **Populate Data**: Update all `TenantId` to the Default Tenant ID.
    ```sql
    UPDATE Products SET TenantId = '11111111-1111-1111-1111-111111111111';
    UPDATE Users SET TenantId = '11111111-1111-1111-1111-111111111111';
    -- Repeat for all tables...
    ```

3.  **Add Constraints**: Set the column to NOT NULL and add foreign keys.
    ```sql
    ALTER TABLE Products ALTER COLUMN TenantId UNIQUEIDENTIFIER NOT NULL;
    ALTER TABLE Products ADD CONSTRAINT FK_Products_Tenants FOREIGN KEY (TenantId) REFERENCES Tenants(Id);
    -- Index Optimization
    CREATE INDEX IX_Products_TenantId ON Products(TenantId);
    ```

### 3.3 Static Resource Migration
If file system storage is used:
1.  Create directory structure: `wwwroot/Tenants/11111111-1111-1111-1111-111111111111/`.
2.  Move existing `wwwroot/uploads` or other resource folders into the above directory.
3.  Run scripts to update file paths in the database (if relative paths are stored, updates may be needed; if filenames are stored, only code-level modification to `PathHelper` is required).

---

## 4. Application Deployment

1.  Deploy the latest API binary files.
2.  Update `appsettings.json` to ensure the connection string is correct.
3.  Restart IIS / Kestrel service.

---

## 5. Verification and Rollback

### 5.1 Verification Steps
1.  **Login Verification**: Log in using an existing administrator account, confirm successful token retrieval, and that the Token contains `TenantId`.
2.  **Data Visibility**: Access product lists, order lists, and confirm original data is visible.
3.  **New Tenant Test**:
    *   Manually insert a new Tenant in the database.
    *   Create a user belonging to the new Tenant.
    *   Log in with the new user and confirm that default tenant data is NOT visible (lists should be empty).

### 5.2 Rollback Plan
If verification fails:
1.  Stop Web service.
2.  Restore database backup.
    ```sql
    RESTORE DATABASE [POSDb] FROM DISK = 'C:\Backups\POSDb_PreMigration.bak' WITH REPLACE;
    ```
3.  Restore old version application files.
4.  Restore static resource file locations.
5.  Restart service.

---

## 6. Frequently Asked Questions (FAQ)

*   **Q: Login failed with "User not found"**
    *   A: Check if `TenantId` in `Users` table is correctly populated and matches the tenant context in the login request.

*   **Q: Performance degradation**
    *   A: Check if indexes have been created on `TenantId` for all tables. EF Core Query Filter adds `WHERE TenantId = ...` to every query; missing indexes will cause full table scans.
