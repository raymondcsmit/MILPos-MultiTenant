# Revised Hybrid Data Access Strategy: EF Core & Dapper Migration Plan

## 1. Executive Summary
This document defines a concrete, step-by-step migration plan for adopting a hybrid data access strategy (Entity Framework Core + Dapper) within the POS & Inventory Management system. 
While EF Core provides excellent developer velocity, tenant isolation, and transaction management for CUD (Create, Update, Delete) operations, profiling has identified severe performance bottlenecks (e.g., >14-second latencies) on complex, read-heavy dashboard aggregations due to SQL translation failures and client-side evaluation. 
This strategy enables surgical, as-needed migration of specific QueryHandlers to Dapper to achieve 2x–5x latency reductions, while strictly preserving multi-tenant safety, multi-database support (SQLite, PostgreSQL, SQL Server), and the existing MediatR caching pipeline.

## 2. Selection Criteria for Dapper Migration
Do **not** migrate all queries to Dapper. Use EF Core (with `.AsNoTracking()`) for simple CRUD reads, paginated list lookups, and low-volume queries.

A QueryHandler qualifies for Dapper migration **only** if it meets the following criteria:
1. **Read-Only Operation**: The handler performs no state mutations (no `SaveChanges`).
2. **Performance Threshold**: The average response time is **> 200 ms** in production or staging under load.
3. **Complexity / Provider Sensitivity**: The query involves high-cardinality aggregations (e.g., `GroupBy` on date parts) or cross-table reporting joins that EF Core translates poorly or evaluates client-side.
4. **Hot Path**: The endpoint is frequently accessed by end-users (e.g., Dashboard landing pages).

## 3. The 7-Step Migration Workflow
When a QueryHandler is selected for migration, follow this strict, repeatable workflow to ensure zero downtime and zero data leakage.

### Step 1: Profile & Baseline (EF Core)
- Use the existing `ApiAndQueriesProfiler` telemetry to document the baseline p50, p90, and p95 latency.
- Capture the raw SQL generated by EF Core and the memory allocation metrics (if available).

### Step 2: Scaffold the Dapper Implementation
- Use the **Vibe-Coding Dapper Template** (type `// MIGRATE-TO-DAPPER` to trigger snippet generation).
- Write the raw SQL queries tailored for each supported database provider. 
- **CRITICAL**: You must manually append the Tenant and Soft-Delete predicates (`WHERE TenantId = @TenantId AND IsDeleted = 0`) to the raw SQL, as Dapper bypasses EF Core's global query filters.
- Maintain the exact same DTO shape and `IRequest<T>` MediatR parameter contract.

### Step 3: Unit & Integration Testing
- Write tests asserting that the new Dapper handler returns the exact same data shape, count, and values as the EF Core handler.
- Run integration tests against Dockerized instances of PostgreSQL, SQL Server, and SQLite to verify SQL dialect compatibility.

### Step 4: Feature Flag Gating
- Do not replace the EF Core handler immediately. 
- Introduce a per-query feature flag (e.g., `Features:Dapper:DashboardIncomeComparison`) via an `IReadStrategySelector`.
- If the flag is false, the code falls back to the original EF Core logic.

### Step 5: Shadow Traffic & Side-by-Side Monitoring
- Deploy to production with the feature flag enabled.
- Monitor the endpoint for 24 hours. Compare the new latency metrics against the baseline established in Step 1.
- **Rollback Condition**: If error rates exceed **0.5%** or latency regressions exceed **10%**, instantly flip the feature flag to `false` via configuration to restore EF Core functionality without a redeployment.

### Step 6: Code Cleanup (Post-Sprint)
- After one successful sprint (2 weeks) of 100% Dapper traffic with no rollbacks, delete the legacy EF Core code block.
- Remove the feature flag for that specific query.

### Step 7: Documentation & Registry Update
- Update the living **QueryHandler Registry** (see below) to reflect the new state, owner, and performance delta.
- Document the new raw SQL in the centralized query library.
